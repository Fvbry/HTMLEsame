Motivazione: In JavaScript gli oggetti sono manipolati tramite le loro proprietà e metodi (funzioni). Inoltre, le interazioni avvengono spessissimo tramite eventi del DOM (es. onClick) che scatenano script per modificare gli oggetti. Domanda 2: Si consideri la seguente chiamata REST

HTTP / XML
PUT/blog/blog_id/pub/5
<titolo>A new assurance technique for the cloud</titolo>
<autore>Claudio Ardagna</autore>
<corpo>Description</corpo>
  • 1. Se pub con id 5 non esiste, lo crea. VERO
  • 2. Se pub con id 5 non esiste, l'azione eseguita è la stessa che verrebbe eseguita con un'operazione POST. VERO
  • 3. Se pub con id 5 esiste, non ha alcun effetto. FALSO
  • 4. Se pub con id 5 esiste, il risultato è equivalente al risultato ottenuto con un'operazione POST. FALSO
Motivazione: L'operazione PUT è idempotente. Se la risorsa puntata dall'URI (id 5) non esiste, il server può crearla (comportandosi come una POST per la creazione, sebbene l'ID sia imposto dal client). Se esiste, il comando PUT la aggiorna/sostituisce (quindi la 3 è falsa). La 4 è falsa perché un POST su una risorsa esistente tipicamente crea una nuova sottomaschera, non la sostituisce.

Domanda 3: Il semantic markup in HTML5

  • 1. Associa un significato al contenuto. VERO
  • 2. Rende una pagina web comprensibile sia all'uomo che alla macchina. VERO
  • 3. Modifica la struttura di una pagina HTML. FALSO
Motivazione: Il markup semantico di HTML5 (es. tag come <article>, <section>, <header>) non altera visivamente o gerarchicamente la struttura base della pagina in modo diverso da un <div>, ma fornisce metadati e significato ("meaning") a quel blocco di contenuto, rendendolo analizzabile da browser e crawler (macchine).

Domanda 4: Si consideri il seguente codice JavaScript

JAVASCRIPT
<script type="text/javascript">
var stringa_globale = "Claudio";

function fz1 (stringa_locale) {
  return stringa_locale + stringa_globale;
}

function fz2() {
  stringa_globale = "Erika";
  return fz1("Ciao ");
}

alert(fz2());
</script>
  • 1. fz2 stampa "Ciao Claudio". FALSO
  • 2. fz2 stampa "Ciao Erika". VERO
  • 3. fz2 ritorna un errore. FALSO
Motivazione (max 3 righe): Nella funzione fz2, la variabile stringa_globale viene modificata riassegnandole il valore "Erika". Poiché non viene usata la parola chiave var, viene sovrascritta la variabile nello scope globale. Quando fz1 viene richiamata, concatena "Ciao " con il nuovo valore globale, ritornando "Ciao Erika".

Domanda a risposta aperta: Regole CSS in cascata

Nell'ambito del linguaggio CSS, si descrivano le regole di valutazione in cascata.

Soluzione:
La "cascata" (Cascade) è l'algoritmo usato dal browser per risolvere i conflitti quando più regole CSS si applicano allo stesso elemento HTML. Le regole di valutazione seguono un ordine di priorità ben preciso:
1. Importanza e Origine: Hanno priorità assoluta le regole con !important. In secondo luogo si valuta l'origine (le dichiarazioni dell'autore della pagina vincono su quelle di default del browser/user agent).
2. Specificità (Specificity): Se due regole hanno la stessa origine, vince il selettore più specifico. La specificità si calcola assegnando pesi: lo stile inline vince sugli ID, gli ID vincono sulle Classi/Pseudo-classi, e le Classi vincono sui semplici Tag HTML.
3. Ordine sorgente (Source Order): A parità di importanza e specificità, viene applicata l'ultima regola dichiarata in ordine cronologico nel codice (l'ultima letta sovrascrive le precedenti).
