Domanda 1: Si consideri il seguente file XML

XML / JS
<cities>
  <city>
    <name>Milan</name>
    <country>Italy</country>
  </city>
  <city>
    <name>Paris</name>
    <country>France</country>
  </city>
  <city>
    <name>New York</name>
    <country>USA</country>
  </city>
</cities>

Il seguente codice HTML:

<div>
  <b>City:</b> <span id="name"></span><br>
  <b>Country:</b> <span id="country"></span><br>
</div>

E il seguente codice JavaScript:

document.getElementById("name").innerHTML = xmlDoc.getElementsByTagName("country")[1].childNodes[0].nodeValue;
document.getElementById("country").innerHTML = xmlDoc.getElementsByTagName("name")[2].childNodes[0].nodeValue;

Quali delle seguenti risposte verrà visualizzata dalla pagina web?

  • 1. "City: France" e "Country: New York". VERO
  • 2. "City: Italy" e "Country: New York". FALSO
  • 3. "City: Italy" e "Country: Paris". FALSO
Motivazione (max 3 righe): La prima istruzione prende l'elemento "country" di indice 1 (il secondo, ovvero "France") e lo inserisce nello span "name". La seconda istruzione prende l'elemento "name" di indice 2 (il terzo, ovvero "New York") e lo inserisce nello span "country". L'output risultante è "City: France" e "Country: New York".

Domanda 2: Nella modalità di accesso a pagine Web tramite nodi di JavaScript

  • 1. Tutti gli elementi di una pagina, inclusi testo e attributi, sono modellati come nodi. VERO
  • 2. È possibile aggiungere, modificare e rimuovere qualsiasi nodo. VERO
  • 3. Tutti i nodi eccetto la root hanno un solo padre. VERO
  • 4. È possibile manipolare la pagina tramite funzioni quali getElementById. VERO
Motivazione: Tutte le affermazioni descrivono correttamente le proprietà fondamentali del Document Object Model (DOM), che rappresenta il documento come un albero gerarchico manipolabile via JavaScript.

Domanda 3: Il protocollo SOAP

  • 1. Definisce il formato generale di un messaggio e come processarlo. VERO
  • 2. Si basa sullo scambio di messaggi composti di un'unica sezione body. FALSO
  • 3. È un protocollo basato su JSON. FALSO
Motivazione: La 2 è falsa perché un messaggio SOAP è composto da un Envelope che racchiude un Body e, opzionalmente, un Header. La 3 è falsa perché SOAP è un protocollo basato rigidamente ed esclusivamente su XML, non su JSON.

Domanda 4: Si consideri il seguente codice CSS

CSS
div {
  width: 100px;
  height: 100px;
  background: red;
  transition: width 2s, height 2s;
}
div:hover {
  width: 300px;
  height: 300px;
  background: blue;
}
  • 1. È un brano di codice CSS3. VERO
  • 2. Applica una trasformazione al background degli elementi div. FALSO
  • 3. Applica una trasformazione quando avviene un click del mouse sull'elemento div. FALSO
  • 4. Definisce una trasformazione completa in 4 secondi. FALSO
Motivazione:
La 1 è vera (transition è una proprietà di CSS3).
La 2 è falsa: il colore cambia, ma a scatto, perché non c'è una transizione dichiarata per il background (solo per width/height), e soprattutto non viene usata la direttiva CSS transform.
La 3 è falsa: :hover si attiva al semplice passaggio del mouse, non al click (per il click sarebbe :active).
La 4 è falsa: le due transizioni partono simultaneamente e durano 2 secondi in totale (non la somma 2+2).

Domanda 5: HTML5

  • 1. Riduce l'utilizzo degli elementi div nella strutturazione della pagina. VERO
  • 2. Mantiene i controlli sintattici stringenti propri di XHTML. FALSO
  • 3. Aggiunge un livello semantico alla pagina web. VERO
Motivazione: HTML5 ha introdotto tag semantici (come <header>, <footer>, <nav>, <article>) che riducono la necessità di usare div annidati, fornendo così un maggiore livello semantico (vero 1 e 3). La 2 è falsa perché HTML5 ha allentato molte regole rigide di XHTML (es. chiusura obbligatoria, tag in minuscolo), rendendo la sintassi molto più permissiva.

Domanda a Risposta Aperta: Architetture REST (POST vs PUT)

Nell'ambito delle architetture REST, si discutano le principali differenze tra le operazioni POST e PUT.

Soluzione:
Le operazioni POST e PUT sono due dei metodi HTTP fondamentali nelle architetture RESTful, impiegati per inviare dati al server, ma si comportano in modo molto diverso, soprattutto rispetto al concetto di idempotenza.

- POST (Creazione Subordinata): Il metodo POST non è idempotente. Viene impiegato per creare una nuova risorsa all'interno di una collezione (es. inviando dati a /utenti/). È il server a generare un ID univoco per la risorsa e a restituire l'URI finale al client (es. `201 Created` con `Location: /utenti/7`). Se un client esegue tre volte la stessa operazione POST, verranno create tre risorse distinte e duplicate.

- PUT (Sostituzione/Creazione Fissa): Il metodo PUT è idempotente. Viene impiegato per aggiornare (sovrascrivere interamente) una risorsa preesistente presso un URI specifico, oppure per crearla nel caso il client sia a conoscenza a priori dell'ID/URI finale (es. PUT /utenti/7). Dato che è idempotente, se la stessa richiesta PUT viene inviata più volte, lo stato del server non subirà variazioni successive, ma rimarrà lo stesso della prima esecuzione (i dati verranno semplicemente rimpiazzati con se stessi).
