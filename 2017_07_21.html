Domanda 1: WSDL

  • 1. È un protocollo per lo scambio di messaggi tra computer sulla rete Internet. FALSO
  • 2. Definisce l'interfaccia di un servizio Web. VERO
  • 3. Include la specifica di come associare il WSDL con SOAP. VERO
  • 4. Specifica l'indirizzo di binding del servizio. VERO
Motivazione: La 1 è falsa perché WSDL (Web Services Description Language) non è un protocollo di "trasporto" o "scambio" messaggi (quelli sono HTTP, SMTP o SOAP stesso), ma è un linguaggio XML usato per *descrivere* i servizi web. Le opzioni 2, 3 e 4 descrivono esattamente la struttura di un documento WSDL: definisce le operazioni (interfaccia), il protocollo di comunicazione (binding, tipicamente SOAP), e l'URL a cui il servizio risponde (endpoint).

Domanda 2: Il tag <body> di un documento HTML

  • 1. Definisce il contenuto del documento HTML. VERO
  • 2. Contiene informazioni utilizzate dai motori di ricerca per indicizzare il documento HTML. FALSO
  • 3. È il tag contenitore di tutto il documento HTML. FALSO
Motivazione: Il tag <body> contiene esclusivamente la parte visibile/strutturale della pagina (il contenuto). La 2 è falsa perché i metadati per i motori di ricerca si inseriscono nei tag <meta> all'interno della sezione <head>. La 3 è falsa perché il contenitore radice di *tutto* il documento è il tag <html>.

Domanda 3: Si consideri il seguente codice JavaScript

JAVASCRIPT
function exec_f(f, x) {
  return f(x);
}
/* Nota: nel testo originale c'è un refuso "return xxx", 
   logicamente inteso come "return x*x" */
alert(exec_f(new Function ("x", "return x*x"), 3));
  • 1. La funzione alert mostra undefined. FALSO
  • 2. La funzione alert mostra 9. VERO
  • 3. La funzione alert mostra 3. FALSO
Motivazione (max 3 righe): Il costruttore new Function("x", "return x*x") crea una funzione anonima che prende in input un parametro e ne restituisce il quadrato. La funzione exec_f la esegue passandole il valore 3. Poiché 3 elevato al quadrato fa 9, l'alert mostrerà il risultato 9.

Domanda 4: Si consideri il seguente codice HTML/JavaScript

HTML / JS
<p>Hello World!</p>
<p id="today">Today is AWC Examination Day!</p>
<p id="disclaimer"></p>
<p id="lunghezza"></p>

<script>
var myNodeList = document.getElementsByTagName("p");
var temp = myNodeList[1].innerHTML;
document.getElementById("disclaimer").innerHTML = temp;
document.getElementById("lunghezza").innerHTML = myNodeList.length;
document.getElementsByTagName("p")[2].innerHTML = document.getElementById("lunghezza").innerHTML;
</script>
  • 1. Il paragrafo con id="disclaimer" e con id="lunghezza" hanno lo stesso valore. VERO
  • 2. Il paragrafo con id="today" e con id="lunghezza" hanno lo stesso valore. FALSO
  • 3. Il paragrafo con id="disclaimer" ha valore 4. VERO
  • 4. Il paragrafo con id="lunghezza" ha valore "Hello World!". FALSO
Motivazione (Trace dello Script):
1) Ci sono in totale 4 paragrafi (length = 4).
2) La variabile temp prende il contenuto del secondo paragrafo (indice 1), ovvero "Today is...".
3) Il paragrafo "disclaimer" viene riempito con temp ("Today is...").
4) Il paragrafo "lunghezza" viene riempito col numero totale di nodi (4).
5) L'ultima istruzione accede al terzo paragrafo della pagina (indice 2, che è proprio "disclaimer"!) e ne sovrascrive il testo copiandoci dentro il contenuto di "lunghezza". Di conseguenza, alla fine dell'esecuzione, "disclaimer" diventa "4" e "lunghezza" è "4".

Domanda a Risposta Aperta 1: Architetture REST (POST vs PUT)

Nell'ambito delle architetture REST, si discutano le principali differenze tra le operazioni POST e PUT. Si presenti un esempio per ognuna delle operazioni.

Soluzione:
Entrambi i metodi HTTP vengono utilizzati per manipolare le risorse inviando dati al server, ma si differenziano per la loro idempotenza (la garanzia che l'esecuzione multipla produca sempre il medesimo stato).

- POST (Non Idempotente): Si utilizza per *creare* una nuova risorsa subordinata. Con il POST, il client invia i dati a un URI radice/collezione, e delega al server la responsabilità di generare un ID univoco e creare il percorso finale per quella risorsa. Se si invia la stessa richiesta POST tre volte, verranno creati tre oggetti distinti.
Esempio: POST /utenti (invia i dati; il server crea l'utente e risponde assegnandogli un ID, es. `/utenti/5`).

- PUT (Idempotente): Si utilizza per *aggiornare (sostituire interamente)* una risorsa esistente, oppure per *crearla* se il client è autorizzato a decidere il suo URI definitivo. L'idempotenza fa sì che eseguire una richiesta PUT più volte sovrascriverà sempre lo stesso elemento lasciando invariato lo stato del database rispetto alla prima esecuzione.
Esempio: PUT /utenti/5 (aggiorna integralmente l'utente numero 5 sostituendo i vecchi dati con il nuovo blocco JSON inviato nel payload).

Domanda a Risposta Aperta 2: Web Storage vs Cookie

Si discutano in dettaglio local storage e session storage, presentando le principali differenze con i cookie.

Soluzione:
Il Web Storage è un'API introdotta con HTML5 per risolvere i limiti dei classici cookie, consentendo al client di memorizzare dati in locale sotto forma di coppie chiave/valore.

Si divide in due oggetti principali:
1. localStorage: Memorizza i dati senza data di scadenza. Persistono anche dopo la chiusura del browser e il riavvio del sistema, e restano accessibili finché non vengono cancellati esplicitamente via codice (es. localStorage.clear()) o svuotando la cache del browser.
2. sessionStorage: Memorizza i dati esclusivamente per la sessione corrente. Se l'utente chiude quella specifica scheda (tab) o la finestra del browser, tutti i dati vengono eliminati irrecuperabilmente.

Principali differenze con i Cookie:
- Traffico di rete (Performance): I cookie vengono inclusi automaticamente in ogni singola richiesta HTTP inviata al server (rallentando la rete se pesanti), mentre il Web Storage risiede solo sul client e non viene mai inviato in automatico, ma prelevato solo quando serve via JavaScript.
- Spazio di archiviazione: I cookie hanno un limite molto stringente (circa 4 KB), mentre il Web Storage permette di salvare una mole di dati notevole (di solito almeno 5 MB per dominio).
