Domanda 1: Si consideri il seguente codice JavaScript

JAVASCRIPT
function exec_f (f,x) {
  return f(x);
}
alert(exec_f(new Function("x", "return x/2"), 14));
  • 1. La funzione alert mostra un errore. FALSO
  • 2. La funzione alert mostra 7. VERO
  • 3. La funzione alert mostra 14. FALSO
Motivazione (max 3 righe): Il costruttore new Function("x", "return x/2") genera dinamicamente una funzione anonima che accetta un parametro x e ne restituisce la metà. Quando viene invocata passandole il valore 14 tramite la funzione exec_f, l'operazione matematica eseguita è 14 diviso 2, che restituisce 7.

Domanda 2: Nella virtualizzazione nativa

  • 1. Si utilizza un unico sistema operativo. FALSO
  • 2. Si ottengono prestazioni migliori. VERO
  • 3. L'hypervisor è ospitato da un sistema operativo host. FALSO
  • 4. L'hypervisor è direttamente installato sull'hardware. VERO
Motivazione: La virtualizzazione nativa (o hypervisor di tipo 1 / bare-metal) prevede che l'hypervisor venga installato direttamente sull'hardware del server (la 4 è vera), senza la presenza di un sistema operativo host sottostante (la 3 è falsa). Questo permette di ospitare molteplici sistemi operativi guest (la 1 è falsa) con un overhead ridotto, garantendo prestazioni decisamente superiori rispetto agli hypervisor ospitati (la 2 è vera).

Domanda 3: Il seguente brano di codice

CSS
#rcorners1 {
  border-radius: 25px/100px 0px 0px 0px;
  background: #8AC007;
  padding: 20px;
  width: 200px;
  height: 150px;
}
  • 1. La proprietà padding definisce la distanza tra contenuto e bordo dell'elemento a cui viene applicato. VERO
  • 2. Può essere specificato all'interno di un file con estensione .css. VERO
  • 3. Definisce un selettore class. FALSO
  • 4. Se applicato a un elemento HTML paragrafo, produce una box contenitore con 4 angoli uguali. FALSO
Motivazione: La 3 è falsa perché il prefisso # identifica in CSS un selettore di id, non di classe (che userebbe il punto). La 4 è falsa perché la regola border-radius specificata ha valori diversi per i 4 angoli (solo il primo angolo in alto a sinistra è arrotondato in modo ellittico con 25px/100px, gli altri 3 angoli sono a spigolo vivo a 0px).

Domanda 4: Un browser

  • 1. È un programma per lo sviluppo di web service. FALSO
  • 2. Interpreta pagine HTML. VERO
  • 3. Permette di interagire con applicazioni web e cloud. VERO
Motivazione: Un browser non serve a sviluppare web service (attività fatta tramite IDE o editor di codice e backend), ma è il client per eccellenza progettato per ricevere documenti web, interpretarne il markup (HTML) e gli script (JS), e permettere all'utente di interfacciarsi visivamente con i servizi cloud.

Domanda 5: Si consideri il seguente brano di codice XML

XML / SOAP
<env:Envelope xmlns:env="http://www.w3.org/2003/05/soap-envelope">
  <env:Body>
    <m:getLastResult xmlns:m="http://example.org/result">
      <m:team>Los Angeles Lakers</m:team>
    </m:getLastResult>
  </env:Body>
</env:Envelope>
  • 1. È parte di una specifica di interfaccia WSDL. FALSO
  • 2. È parte di una richiesta SOAP. VERO
  • 3. Viene usato da un web service che definisce interfacce standard HTTP. FALSO
Motivazione: I tag Envelope e Body sono i costrutti fondamentali per il payload di un messaggio SOAP (la 2 è vera). La 1 è falsa perché il WSDL serve a descrivere il contratto del servizio, non a comporre la singola richiesta con i dati reali ("Los Angeles Lakers"). La 3 è falsa perché un web service basato su SOAP definisce metodi personalizzati incapsulati nel body (come getLastResult) piuttosto che usare interfacce a risorse basate rigorosamente sui metodi standard HTTP (tipico invece del REST).

Domanda 6: XML vs JSON

  • 1. XML è usato per descrivere dati strutturati che non includono array. VERO
  • 2. Il metodo eval di JavaScript fa il parsing di documenti XML. FALSO
  • 3. JSON è più verboso di XML. FALSO
Motivazione: La 1 è vera: il linguaggio XML nativamente non possiede il tipo di dato "Array" (si simula semplicemente ripetendo lo stesso tag più volte), mentre JSON sì. La 2 è falsa perché la storica e deprecata funzione eval() valuta stringhe come codice JavaScript (o JSON); per parsare XML in JS serve il DOMParser. La 3 è palesemente falsa poiché JSON, privo di tag di chiusura, è molto più leggero e meno verboso di XML.

Domanda a Risposta Aperta: Local storage e Session storage

Si discutano in dettaglio local storage e session storage, presentando le loro principali differenze.

Soluzione:
Local Storage e Session Storage costituiscono le due API fondamentali dello standard Web Storage introdotto da HTML5. Il loro scopo è permettere alle applicazioni web di memorizzare dati in locale sul browser dell'utente in un formato a coppie chiave/valore (Key-Value), in modo molto più efficiente e capiente rispetto ai tradizionali Cookie (i dati non vengono inviati automaticamente al server con ogni singola richiesta HTTP).

Le principali differenze riguardano la durata (lifetime) e la visibilità:
- window.localStorage: Salva i dati in modo persistente e duraturo. Un dato salvato in local storage non possiede alcuna data di scadenza. Persiste anche se l'utente chiude completamente il browser, spegne il computer e riapre il sito giorni dopo. Può essere eliminato solo in maniera esplicita tramite codice JS (localStorage.clear() o removeItem()) oppure se l'utente svuota volontariamente i dati/cache del browser.
- window.sessionStorage: Salva i dati in maniera volatile ed esclusivamente per la durata della singola sessione della pagina attiva. Se l'utente chiude la specifica scheda (tab) del browser o la singola finestra, i dati archiviati nel session storage per quell'origine vengono automaticamente e definitivamente cancellati. È ideale per salvare dati di input provvisori o per mantenere lo stato all'interno della navigazione di quel tab.
