Domanda 1: Si consideri il seguente codice JavaScript

JAVASCRIPT
<script language="javascript">
var nome = "Claudio";
function fz1 (cognome) {
  return nome + " " + cognome;
}
function fz2() {
  var nome = "Denise";
  return fz1("Ardagna");
}
window.alert(fz2());
</script>
  • 1. fz2 stampa "Claudio Ardagna". VERO
  • 2. fz2 stampa "Denise Ardagna". FALSO
  • 3. fz2 ritorna un errore. FALSO
Motivazione (max 3 righe): In JavaScript vige lo scope lessicale. La funzione fz1 fa riferimento alla variabile nome presente nel contesto in cui è stata dichiarata, ovvero la variabile globale "Claudio". Anche se fz2 dichiara una variabile locale omonima ("Denise"), fz1 la ignora e continua ad usare quella globale.

Domanda 2: Il selettore id di CSS

  • 1. Identifica lo stile di un singolo elemento all'interno della pagina HTML. VERO
  • 2. Permette di scegliere il nome del selettore da un vocabolario fisso di parole. FALSO
  • 3. È definito in un file CSS tramite la sintassi "#nomeselettore". VERO
  • 4. Può essere combinato con selettori class. VERO
Motivazione: La 2 è falsa perché i nomi degli ID (così come le classi) sono completamente arbitrari e decisi dallo sviluppatore. La 4 è vera: è perfettamente valido in CSS combinare un id e una classe per aumentare la specificità (es. #mioHeader.active { color: red; }, che colpirà l'elemento con quell'ID solo se possiede anche la classe active).

Domanda 3: JSON notation

  • 1. Viene gestito in JavaScript tramite le funzioni JSON.stringify e JSON.parse. VERO
  • 2. Permette di formattare strutture dati. VERO
  • 3. È più leggera di XML. VERO
Motivazione: Tutte e tre le affermazioni sono corrette. JSON (JavaScript Object Notation) nasce appositamente per strutturare e scambiare dati (2) risultando molto meno verboso di XML non avendo la necessità di tag di chiusura (3). Il parsing e la serializzazione nativa in JS si effettuano rispettivamente con JSON.parse() e JSON.stringify() (1).

Domanda 4: Si consideri la seguente chiamata REST

HTTP / REST
PUT /sondaggio/id_sondaggio/voto/7
<nome>Claudio</nome>
<nazione>Italia</nazione>
<opzione>7</opzione>
  • 1. Se il voto con id 7 non esiste lo crea. VERO
  • 2. Se il voto con id 7 non esiste il risultato è equivalente al risultato ottenuto con un'operazione POST. VERO
  • 3. Se il voto con id 7 esiste non ha alcun effetto. FALSO
  • 4. Se il voto con id 7 esiste il risultato è equivalente al risultato ottenuto con un'operazione POST. FALSO
Motivazione: Il metodo PUT in un servizio REST può essere usato sia per creare una risorsa (se il client è autorizzato a stabilirne a priori l'URI, in questo caso ".../7") sia per aggiornarla (sovrascrivendola) nel caso esistesse già. L'affermazione 3 è quindi falsa perché, se esiste, il PUT ne sostituisce lo stato. La 4 è falsa perché un POST su una collezione pre-esistente generalmente ne crea un'altra nuova (non è idempotente) mentre il PUT è sempre idempotente.

Domanda a Risposta Aperta 1: XML vs JSON

Dopo aver descritto le caratteristiche del linguaggio XML nel dettaglio e presentato un esempio completo di file XML, si discutano le principali differenze con JSON.

Soluzione:
L'XML (eXtensible Markup Language) è un metalinguaggio che permette di definire altri linguaggi di marcatura. Le sue caratteristiche principali sono l'estensibilità (lo sviluppatore crea i propri tag, es. <utente>), la strutturazione gerarchica ad albero, e la rigidità sintattica (ogni tag aperto deve essere chiuso, ci deve essere un solo elemento radice). È molto utilizzato nello scambio di messaggi aziendali e configurazioni.

Esempio XML: 
<?xml version="1.0" encoding="UTF-8"?>
<libreria>
  <libro categoria="romanzo">
    <titolo>1984</titolo>
    <autore>George Orwell</autore>
  </libro>
</libreria>

Differenze principali con JSON:
- Verbosità: XML richiede tag di chiusura continui (es. <titolo>...</titolo>) risultando molto più pesante da trasferire su rete rispetto a JSON, che usa solo parentesi graffe, quadre e virgolette (chiave-valore).
- Tipi di dato: XML tratta tutto nativamente come stringa testuale, mentre JSON supporta in modo nativo numeri, booleani e soprattutto la struttura dati ad Array ([]).
- Parsing: In ambiente Web, il parsing di JSON in oggetti Javascript è immediato (JSON.parse), mentre XML necessita di parser DOM dedicati che richiedono codice molto più complesso per l'attraversamento dei nodi.
- Metadati: XML supporta gli attributi all'interno del tag di apertura per gestire metadati, cosa che in JSON non esiste in modo nativo (tutto è semplicemente proprietà dell'oggetto).

Domanda a Risposta Aperta 2: Web Storage

Si discutano in dettaglio local storage e session storage, presentando le principali differenze.

Soluzione:
Il Web Storage (definito comunemente in HTML5) fornisce API JavaScript per l'archiviazione di dati lato client sotto forma di coppie chiave/valore, superando i pesanti limiti di spazio e le inefficienze di rete imposti dai classici cookie.

Si compone di due oggetti globali del browser:
1. window.localStorage: Salva i dati in modo persistente. Questo significa che i dati archiviati qui non hanno alcuna data di scadenza. Se l'utente chiude il browser, spegne il computer e riapre il sito il giorno dopo, i dati (come preferenze di tema o carrelli salvati) saranno ancora lì disponibili.
2. window.sessionStorage: Salva i dati in un'area di memoria legata strettamente alla singola sessione di navigazione della pagina. Se l'utente chiude quella specifica scheda (tab) o la finestra del browser, tutti i dati archiviati nel sessionStorage vengono immediatamente e permanentemente distrutti.

Differenza principale: La durata o persistenza del ciclo di vita del dato (Lifetime). SessionStorage è volatile e muore con la chiusura del tab; LocalStorage è permanente e muore solo se cancellato esplicitamente via codice (localStorage.removeItem()) o se l'utente svuota la cache/memoria locale del proprio browser.
