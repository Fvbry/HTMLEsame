Domanda 1: Si consideri il seguente codice HTML/JavaScript

HTML / JS
<div class="song">
  <button onclick="hide('song_1')">x</button>
  <div class="description" id="song_1">
    <!-- Contenuto omesso per brevità -->
  </div>
</div>
<script>
function hide (target) {
  elem=document.getElementById(target);
  if (elem.style.display=="none")
    elem.style.display="block";
  else
    elem.style.display="none";
}
</script>
  • 1. La funzione hide viene eseguita al caricamento della pagina HTML. FALSO
  • 2. La funzione hide modifica una proprietà CSS. VERO
  • 3. elem.style.display="none" elimina l'elemento elem dalla pagina HTML lasciando tuttavia lo spazio occupato originariamente da elem. FALSO
  • 4. document.getElementById(target) assegna alla variabile elem l'elemento con attributo id="target". VERO
Motivazione: La 1 è falsa perché la funzione si attiva solo al click del bottone (onclick), non all'avvio. La 3 è falsa perché display: none rimuove completamente l'elemento dal flusso della pagina, compreso il suo spazio (è visibility: hidden che invece lascia lo spazio intatto).

Domanda 2: CSS box model

  • 1. Racchiude tutti gli elementi HTML in un box. VERO
  • 2. Specifica unicamente margin, border, padding. FALSO
  • 3. Permette di definire stile, spessore e colore del border. VERO
  • 4. Permette di definire formati diversi per i quattro lati della box. VERO
Motivazione: La 2 è falsa perché il box model non specifica "unicamente" quelli, ma include anche e soprattutto il content (area del contenuto, gestita con width e height). La 4 è vera: si possono definire lati specifici (es. margin-top, border-left-style, ecc.).

Domanda 3: Il tag <body> di un documento HTML

  • 1. Definisce il contenuto del documento HTML. VERO
  • 2. Può contenere brani di codice CSS. VERO
  • 3. Contiene il titolo della pagina/scheda. FALSO
Motivazione: Il <body> contiene gli elementi visibili della pagina. Può contenere codice CSS tramite l'attributo style="" (stile inline) o tag <style> (consentiti da specifiche HTML5 anche nel body). La 3 è falsa perché il titolo della scheda è definito dal tag <title>, che va posizionato obbligatoriamente nella sezione <head>.

Domanda 4: REpresentational State Transfer (REST)

  • 1. Si riferisce a una risorsa tramite URL. VERO
  • 2. Una stessa risorsa può essere rappresentata con formati diversi. VERO
  • 3. Utilizza un'interfaccia uniforme. VERO
  • 4. Utilizza il protocollo FTP per le comunicazioni. FALSO
Motivazione: La 4 è palesemente falsa: le architetture REST sono basate quasi esclusivamente sul protocollo HTTP, sfruttandone i metodi standard (GET, POST, PUT, DELETE). Non si usa FTP per gestire API RESTful.

Domanda 5: Si consideri il seguente codice CSS

CSS
#target {
  position: relative;
  padding: 10px;
  color: white;
  background-color: #b56457;
  height: 80px;
  width: 98%;
  display: inline-block;
}
  • 1. La proprietà padding definisce la distanza tra contenuto e bordo dell'elemento a cui viene applicato. VERO
  • 2. È un brano di codice HTML. FALSO
  • 3. Definisce lo stile di visualizzazione per tutti gli elementi con class="target". FALSO
  • 4. Può essere specificato nel tag <body> di un documento HTML. FALSO
Motivazione: La 2 è falsa perché è codice CSS. La 3 è falsa in quanto il simbolo # identifica un id (es. id="target"), mentre per una classe il selettore avrebbe usato il punto (.target). La 4 è da considerarsi falsa nella pratica classica: le definizioni a blocco del CSS come questa vanno racchiuse in un tag <style> che di norma risiede nell'<head> (oppure in un file .css esterno).

Domanda a Risposta Aperta 1: Rappresentazione JSON

Si consideri il seguente codice XML. Si proponga una rappresentazione equivalente basata su JSON discutendo tutte le scelte effettuate.

XML
<CATALOG>
  <CD>
    <TITLE>Widt om natten</TITLE>
    <ARTIST>Kim Larsen</ARTIST>
    <COUNTRY>EU</COUNTRY>
    <COMPANY>Medley</COMPANY>
    <PRICE>7.90</PRICE>
    <YEAR>1983</YEAR>
  </CD>
  <!-- Altri elementi CD omessi -->
</CATALOG>
Soluzione e Discussione:
{
  "catalog": [
    {
      "title": "Widt om natten",
      "artist": "Kim Larsen",
      "country": "EU",
      "company": "Medley",
      "price": 7.90,
      "year": 1983
    }
  ]
}
Scelte effettuate:
1. Array per elementi multipli: L'elemento radice CATALOG racchiude vari CD. In JSON, questo viene tradotto perfettamente assegnando alla chiave "catalog" un array ([]) di oggetti, visto che un array rappresenta nativamente una lista di entità.
2. Struttura ad Oggetti: Ogni singolo CD è stato mappato come un oggetto JSON ({}).
3. Tipizzazione nativa: In XML tutti i dati sono testuali. In JSON ho tradotto price in un valore decimale (7.90) e year in un numero intero (1983) rimuovendo le virgolette, sfruttando così la capacità di JSON di gestire tipi di dato numerici, rendendo le eventuali manipolazioni JS molto più dirette.

Domanda a Risposta Aperta 2: Web Storage vs Cookie

Si discutano in dettaglio local storage e session storage, presentando le principali differenze con i cookie.

Soluzione:
Local Storage e Session Storage formano l'API Web Storage introdotta da HTML5, ideata per immagazzinare dati strutturati (chiave-valore) nel browser dell'utente in modo molto più efficiente dei Cookie.

- Local Storage: Salva i dati in modo persistente senza una data di scadenza. I dati restano salvati anche se l'utente chiude il browser o riavvia il computer, e devono essere cancellati manualmente (via JS o svuotando la cache del browser).
- Session Storage: Salva i dati per la sola durata della sessione corrente. Se l'utente chiude la specifica scheda (tab) del browser in cui il sito era aperto, i dati vengono persi immediatamente.

Differenze con i Cookie:
1. Capacità: I cookie possono contenere pochissimi dati (circa 4KB), mentre le API Web Storage permettono di salvare almeno 5MB di dati per dominio.
2. Traffico di Rete: I cookie vengono inclusi in modo automatico nell'header di ogni singola richiesta HTTP che il client fa al server, rallentando la rete se sono pesanti. I dati del Web Storage rimangono unicamente sul client (il browser) e non vengono mai inviati al server in automatico.
3. API: L'interfaccia JS per manipolare i cookie è arcaica, mentre Web Storage fornisce metodi moderni e diretti come setItem() e getItem().
