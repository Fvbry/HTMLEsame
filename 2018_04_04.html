Domanda 1: Si consideri il seguente codice JavaScript

JAVASCRIPT
<script type="text/javascript">
var metadati = [
  {"isbn":"isbn1", "prezzo": "20euro"},
  {"isbn":"isbn2", "prezzo": "30euro"},
  {"isbn":"isbn3", "prezzo": "15euro"},
  {"isbn":"isbn4", "prezzo": "50euro"}
];
var libro = [
  {"isbn":"isbn1", "edizione": 10, "titolo": "Anna Karenina", "autore": "Leo Tolstoy"},
  {"isbn":"isbn2", "edizione": 3, "titolo": "Madame Bovary", "autore": "Gustave Flaubert"},
  {"isbn":"isbn3", "edizione": 7, "titolo": "Hamlet", "autore": "William Shakespeare"},
  {"isbn":"isbn4", "edizione": 5, "titolo": "Divina Commedia", "autore": "Dante Alighieri"}
];

localStorage.setItem("md", JSON.stringify(metadati));
localStorage.setItem("book", JSON.stringify(libro));

var chiave = localStorage.key(0);
var tmp = JSON.parse(localStorage.getItem(chiave));

for (i=0; i<metadati.length; i++)
  if (metadati[i].isbn == tmp[2].isbn)
    alert(tmp[2].prezzo);
</script>
  • 1. alert stampa 20euro FALSO
  • 2. alert stampa 30euro FALSO
  • 3. alert stampa 15euro VERO (didatticamente)
  • 4. alert stampa undefined VERO (tecnicamente)
Motivazione (max 3 righe): Domanda trabocchetto! Nelle API Web Storage (localStorage.key(0)) l'ordine delle chiavi non è garantito. Se il browser le ordina alfabeticamente, "book" viene prima di "md", quindi tmp è `libro`. L'oggetto `libro` non ha la proprietà `prezzo`, rendendo l'output undefined (Opzione 4). Se invece il docente ipotizza un ordine basato sull'inserimento (o se "md" finisce in posizione 0), tmp è `metadati`, il cui elemento all'indice 2 ("isbn3") ha prezzo "15euro" (Opzione 3). Ti consiglio di tenere a mente la "3" per la logica dell'esercizio, ma l'effettivo output in molti browser è undefined.

Domanda 2: La funzionalità drag&drop

  • 1. Permette di spostare qualsiasi elemento in un'altra posizione VERO
  • 2. È parte di HTML5 VERO
  • 3. Utilizza l'attributo draggable che con valore true rende un elemento trascinabile VERO
Motivazione: La funzionalità Drag and Drop è stata standardizzata in HTML5 e permette, tramite l'attributo nativo draggable="true", di prendere praticamente qualsiasi elemento HTML della pagina (immagini, div, testo) e trascinarlo in specifiche zone definite come Drop Zone (gestendo gli eventi JS).

Domanda 3: JavaScript permette accesso e modifica di oggetti tramite

  • 1. Oggetti basati su proprietà e funzioni VERO
  • 2. Eventi scatenati da azioni degli utenti VERO
  • 3. Formule booleane FALSO
  • 4. Funzioni HTML FALSO
Motivazione: In JavaScript gli oggetti vengono manipolati accedendo alle loro proprietà (es. `obj.colore`) o richiamando le loro funzioni/metodi. Tali modifiche avvengono quasi sempre in risposta ad eventi (es. click del mouse, caricamento). Le funzioni appartengono ai linguaggi di programmazione (JS), non all'HTML, che è un linguaggio di solo markup.

Domanda 4: La regola @font-face

  • 1. Permette di includere solo alcuni tipi di carattere FALSO
  • 2. Definisce il set di caratteri da includere VERO
  • 3. Permette di selezionare un solo set di caratteri per volta FALSO
  • 4. È disponibile in CSS3 VERO
Motivazione: La 1 è falsa perché supporta innumerevoli formati di web font (TTF, OTF, WOFF, WOFF2). La 3 è falsa perché all'interno del foglio di stile CSS possono coesistere decine di regole @font-face diverse per caricare svariati font (es. un font per i titoli e uno per i paragrafi).

Domanda 5: Si consideri il seguente brano di codice

HTTP / REST
POST /book
201 CREATED
Location: /book/{id}
  • 1. È parte di una comunicazione verso un servizio SOAP FALSO
  • 2. Comporta la creazione di un nuovo book con id selezionato dal server VERO
  • 3. Fornisce lo stesso risultato di una chiamata "PUT /book/{id}" FALSO
  • 4. Utilizza API fatte su misura per il caso specifico FALSO
Motivazione: L'uso dei metodi HTTP standard in combinazione con i codici di stato (come "201 CREATED") e gli Header HTTP ("Location") è il marchio di fabbrica di un'API RESTful, non SOAP (che userebbe XML). La 2 è vera: il metodo POST crea una nuova risorsa subordinata e delega al server il compito di deciderne l'URI/ID (ritornato poi nell'header). La 3 è falsa perché il metodo PUT, oltre a essere idempotente, richiederebbe che il client conoscesse e definisse l'ID in anticipo.

Domanda a Risposta Aperta: XML vs JSON

Si descriva nel dettaglio le caratteristiche di XML, discutendo almeno uno scenario applicativo e le differenze con JSON.

Soluzione:
L'XML (eXtensible Markup Language) è un meta-linguaggio di marcatura progettato per descrivere e strutturare documenti di dati in formato testuale in modo che siano leggibili sia dalle macchine che dagli umani. Le sue caratteristiche principali sono la struttura gerarchica ad albero, l'uso di tag di apertura e chiusura personalizzabili dallo sviluppatore (es. <libro>), la possibilità di aggiungere metadati tramite gli attributi nei tag e l'estrema rigorosità (un file non ben formato blocca l'esecuzione).

Scenario applicativo: L'XML è storicamente lo standard per lo scambio di messaggi complessi in ambito enterprise attraverso i Web Service SOAP, e viene impiegato massicciamente nei file di configurazione software (es. i file pom.xml di Maven o web.xml in Java EE).

Differenze con JSON: JSON (JavaScript Object Notation) è un formato più moderno, notevolmente più leggero (meno verboso) perché privo dei continui tag di chiusura di XML. JSON supporta nativamente gli array ([]) e i tipi di dato nativi (numeri, booleani), cosa che XML non fa (in XML è tutto testo). Infine, JSON è integrato nativamente in JavaScript, rendendo il suo parsing in ambienti Web/Browser infinitamente più veloce rispetto alla scansione ad albero richiesta per il DOM XML.
