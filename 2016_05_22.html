Domanda 1: Si consideri il seguente codice JavaScript

HTML / JAVASCRIPT
<FORM NAME="modulo" onSubmit="print(); return false;">
  <SELECT NAME="menu" id="menuid">
    <OPTION ID="op1" VALUE="Paris">Paris</OPTION>
    <OPTION ID="op2" VALUE="rome">Rome</OPTION>
  </SELECT>
  <INPUT TYPE="Submit" VALUE="OK">
</FORM>

<SCRIPT language="JavaScript">
var city = prompt("Are you European or American?", "European");
if(city == "American") {
  window.document.modulo.menu.options[0].text="New York";
  window.document.modulo.menu.options[1].text="San Francisco";
  var select= document.getElementById('menuid');
  var opt= document.createElement('option');
  opt.id= "op3";
  opt.value= "Cleveland";
  opt.innerHTML= "Cleveland";
  select.appendChild (opt);
}
</SCRIPT>

La pagina HTML mostra un menù a tendina con:

  • 1. Paris e Rome sempre presenti tra le opzioni. FALSO
  • 2. New York in posizione 0 ove presente. VERO
  • 3. Cleveland e Paris mai insieme. VERO
  • 4. Cleveland sempre dopo Paris. FALSO
Motivazione (max 6 righe): Se l'utente risponde "American", i testi di Paris (indice 0) e Rome (indice 1) vengono modificati in "New York" e "San Francisco", e viene aggiunto "Cleveland" in coda. Pertanto, Paris e Rome non sono sempre presenti (1 è Falsa). New York compare in posizione 0 solo se la condizione è soddisfatta (2 è Vera). Cleveland viene creata solo quando Paris viene eliminata/sovrascritta, quindi non coesistono mai (3 è Vera, 4 è Falsa).

Domanda 2: Il seguente brano di codice

JAVASCRIPT
xmlhttp=new XMLHttpRequest();
xmlhttp.open("GET", url, true);
xmlhttp.send();
xmlDoc=xmlhttp.responseXML;
  • 1. Permette di scambiare dati con il server in background. VERO
  • 2. Invia una richiesta HTTP GET al server. VERO
  • 3. È supportato da tutti i browser moderni. VERO
  • 4. Funzionerebbe correttamente con una url che punta a un file testuale diverso da XML. FALSO
Motivazione: Le prime tre affermazioni descrivono correttamente le funzionalità asincrone standard dell'oggetto `XMLHttpRequest` con metodo GET. La 4 è falsa perché l'ultima istruzione estrae esplicitamente `responseXML`; se la risorsa puntata non fosse un file XML valido e ben formato, la proprietà restituirebbe `null` inficiando il corretto funzionamento logico dello script.

Domanda 3: La funzionalità drag&drop

  • 1. Rende un elemento trascinabile tramite l'attributo "draggable=true". VERO
  • 2. Permette di trascinare solo elementi testuali. FALSO
  • 3. Era disponibile prima di HTML5. FALSO
Motivazione: Il Drag and Drop è una specifica nativa introdotta e standardizzata esclusivamente a partire da HTML5 (la 3 è falsa). Essa consente il trascinamento non solo di testo, ma di qualsiasi tipo di nodo dell'albero DOM, inclusi blocchi complessi, immagini e file esterni (la 2 è falsa).

Domanda 4: Si consideri il seguente brano di codice

HTTP / REST
POST /hotel_room
201 CREATED
Location: /hotel_room/{id}
  • 1. È parte di una comunicazione verso un servizio REST. VERO
  • 2. Comporta la creazione di una nuova hotel_room con id specificato dal richiedente. FALSO
  • 3. Fornisce lo stesso risultato di una chiamata "PUT/hotel_room/{id}". FALSO
Motivazione: L'uso combinato di verbi HTTP standard, codici di stato di creazione (201) e l'header `Location` identifica univocamente lo stile architetturale REST (la 1 è vera). La 2 è falsa perché con il metodo `POST` l'identificatore viene generato ed assegnato autonomamente dal server, non dal richiedente. La 3 è falsa perché `PUT` richiede l'ID direttamente nell'URI della chiamata ed è un'operazione idempotente di sovrascrittura completa.

Domanda 5: Il selettore class di CSS

  • 1. Mappa a uno specifico tag. FALSO
  • 2. È definito in un file css tramite la sintassi ".nomeselettore". VERO
  • 3. È associato a un tag HTML tramite l'attributo class. VERO
Motivazione: L'affermazione 1 identifica il selettore di tipo (o di tag), mentre il selettore di classe è slegato da un tag specifico e può essere applicato trasversalmente a qualunque elemento HTML della pagina (la 1 è falsa). Nel foglio di stile si dichiara anteponendo il punto `.` al nome (la 2 è vera) e si collega agli elementi usando la proprietà `class="..."` (la 3 è vera).

Domanda a risposta aperta

Si descriva in dettaglio lo standard JSON e si discutano le principali differenze con lo standard XML.

Soluzione:
Il **JSON** (JavaScript Object Notation) è un formato leggero per lo scambio di dati basato su testo, indipendente dal linguaggio ma che eredita la struttura letterale degli oggetti di JavaScript. Organizza i dati esclusivamente secondo due strutture: collezioni di coppie chiave/valore (oggetti racchiusi in `{}`) ed elenchi ordinati di valori (array racchiusi in `[]`). Supporta nativamente tipi di dato fondamentali quali stringhe, numeri, booleani e il valore `null`.

**Differenze principali con XML:**
  • **Verbosità e Peso:** XML utilizza tag di apertura e chiusura espliciti per ogni dato, risultando significativamente più prolisso, ridondante e pesante da trasmettere sulla rete rispetto alla notazione compatta di JSON.
  • **Tipi di dato nativi:** XML memorizza e interpreta intrinsecamente ogni valore come stringa di testo puro. JSON introduce la tipizzazione nativa permettendo di distinguere direttamente un numero o un valore booleano senza ricorrere a successivi casting software.
  • **Supporto agli Array:** JSON possiede una sintassi nativa immediata per la dichiarazione dei vettori/liste (`[]`). In XML non esistono array nativi, costringendo lo sviluppatore ad annidare tag ripetitivi con lo stesso nome.
  • **Parsing ed Integrazione Web:** In ambiente browser, l'integrazione di JSON con JavaScript è immediata e velocissima grazie al metodo nativo `JSON.parse()`. XML richiede invece l'interrogazione dell'albero del DOM tramite parser più pesanti e l'uso di routine complesse di navigazione tra nodi.
