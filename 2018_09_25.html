Domanda 1: Si consideri il seguente codice JSON

JSON
{
  "car": [
    {
      "Name": "chevrolet chevelle malibu",
      "Miles_per_Gallon": 18,
      "Cylinders": 8,
      "Displacement": 307,
      "Horsepower": 130,
      "Origin": "USA",
      "Available": ["USA", "Europe"]
    },
    {
      "Name": "toyota corona mark ii",
      "Miles_per_Gallon": 24,
      "Cylinders": 4,
      "Displacement": 113,
      "Horsepower": 95,
      "Origin": "Japan",
      "Available": ["Asia", "Europe"]
    },
    {
      "Name": "mazda glc",
      "Miles_per_Gallon": 46.6,
      "Cylinders": 4,
      "Displacement": 86,
      "Horsepower": 65,
      "Origin": "Japan",
      "Available": ["Asia", "USA", "Europe"]
    }
  ]
}
  • 1. Specifica quattro array diversi. FALSO
  • 2. Specifica solo valori di tipo stringa. FALSO
  • 3. Specifica ogni singola car come un array di sette elementi. FALSO
  • 4. Rispetta le regole sintattiche di JSON. VERO
Motivazione: La 1 è falsa perché c'è l'array principale `car` più i 3 array `Available` interni. La 2 è falsa perché contiene valori numerici (es. `18`, `8`, `46.6`). La 3 è falsa perché ogni automobile non è un array, bensì un oggetto (racchiuso tra graffe `{}`) contenente sette proprietà chiave-valore. La 4 è vera: il codice è formalmente un documento JSON valido.

Domanda 2: Si consideri un'applicazione REST per l'aggiunta di un'automobile con il seguente codice

HTTP / REST
HTTP_OPERATION URL
{
  "Name": "chevrolet chevelle malibu",
  "Miles_per_Gallon": 18,
  "Cylinders": 8,
  "Displacement": 307,
  "Horsepower": 130,
  "Origin": "USA",
  "Available": ["USA", "Europe"]
}

N.B. Dopo l'operazione di aggiunta ogni macchina deve avere un id univoco.
Quale delle seguenti affermazioni è vera?

  • 1. HTTP_OPERATION=POST e URL=/garage/{id}/car/{id} FALSO
  • 2. HTTP_OPERATION=PUT e URL=/garage/{id}/car/ FALSO
  • 3. HTTP_OPERATION=POST e URL=/garage/{id}/car/ VERO
Motivazione (max 3 righe): Poiché l'operazione consiste nell'aggiungere una nuova automobile e l'ID univoco non è noto in anticipo (sarà assegnato dal server), si utilizza il metodo `POST` indirizzato alla collezione genitore (la URL terminante con `/car/`).

Domanda 3: XHTML

  • 1. È più stringente di HTML. VERO
  • 2. Viene introdotto per ridurre il numero di pagine web con codice HTML errato. VERO
  • 3. Combina HTML con XML. VERO
  • 4. Ha regole sintattiche meno rigide di HTML. FALSO
Motivazione: XHTML (Extensible HyperText Markup Language) è una riformulazione di HTML come applicazione XML. Proprio per questo motivo, impone regole di sintassi molto più rigide (es. tag chiusi obbligatoriamente, attributi tra virgolette, tag in minuscolo), rendendo l'opzione 4 palesemente falsa.

Domanda 4: XML-RPC

  • 1. Sono gli antenati di SOAP e REST per lo sviluppo di applicazioni distribuite. VERO
  • 2. Non forniscono funzionalità per una comunicazione sicura. VERO
  • 3. Supportano i soli tipi complessi Array e Struct. VERO
  • 4. Utilizzano un protocollo non standard per il trasporto. FALSO
Motivazione: La 4 è falsa perché XML-RPC utilizza HTTP (che è il protocollo standard per eccellenza del web) come meccanismo di trasporto per inviare messaggi XML. Le prime tre affermazioni descrivono correttamente i limiti e le caratteristiche storiche di XML-RPC.

Domanda 5: In JavaScript

  • 1. È possibile definire variabili con due scope: locale e globale. VERO
  • 2. Non è possibile definire procedure. FALSO
  • 3. Le funzioni vengono chiamate fornendo la lista dei parametri attuali. VERO
  • 4. L'accesso e la modifica di oggetti è permesso tramite proprietà e funzioni degli oggetti. VERO
Motivazione: La 2 è falsa perché in programmazione una "procedura" è semplicemente una funzione che non restituisce alcun valore (non ha l'istruzione return o ritorna implicitamente undefined). In JavaScript questo è comunissimo.

Domanda a Risposta Aperta 1: Oggetto XMLHttpRequest

Si discuta in dettaglio il ruolo dell'oggetto XMLHttpRequest presentando un esempio del suo utilizzo.

Soluzione:
L'oggetto XMLHttpRequest (XHR) è il cuore della tecnologia AJAX. Svolge il ruolo fondamentale di scambiare dati (sia XML, che JSON, che testo normale) con un server web in background. Questo permette di aggiornare e ricaricare solo porzioni specifiche di una pagina web senza dover effettuare un refresh (ricaricamento) totale dell'intera pagina, migliorando enormemente l'esperienza utente.

Esempio di utilizzo:
var xhttp = new XMLHttpRequest();
xhttp.onreadystatechange = function() {
  // Se la richiesta è completata (4) e andata a buon fine (200)
  if (this.readyState == 4 && this.status == 200) {
    document.getElementById("demo").innerHTML = this.responseText;
  }
};
xhttp.open("GET", "dati.txt", true); // Imposta la richiesta asincrona
xhttp.send(); // Invia la richiesta al server

Domanda a Risposta Aperta 2: Modellazione XML vs JSON

A partire dal codice JSON della Domanda 1, si discuta quali problematiche si potrebbero affrontare nella modellazione dello stesso file usando XML. Si confrontino inoltre brevemente vantaggi e svantaggi di una modellazione XML e una modellazione JSON.

Soluzione:
Problematiche con XML:
Se dovessimo modellare l'array di stringhe "Available": ["USA", "Europe"] in XML, non avendo una sintassi nativa per gli array, dovremmo annidare tag ripetuti (es. <Available><Region>USA</Region><Region>Europe</Region></Available>). Inoltre, i dati numerici (come Miles_per_Gallon) verrebbero tutti convertiti in testo piano, richiedendo un casting esplicito in fase di lettura da parte del programma.

Vantaggi e Svantaggi (Confronto):
- XML: È molto più verboso, "pesante" per la rete e complicato da parsare lato client (richiede algoritmi di attraversamento del DOM). Il suo grande vantaggio è però l'estensibilità, il supporto ai Namespace, e la possibilità di effettuare validazioni formali molto rigide (tramite file DTD o XSD), rendendolo ideale per servizi di back-end istituzionali (come SOAP).
- JSON: È estremamente leggero, compatto, nativo per JavaScript (si traduce in oggetti JS in un istante con JSON.parse()), e supporta intrinsecamente liste e tipi numerici. Di contro, è privo di un forte meccanismo standard di validazione integrata ed è più difficile inserirvi commenti o metadati complessi.
